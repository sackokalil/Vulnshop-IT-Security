Cross-Site Scripting (XSS)

Security Lab: This page documents intentionally implemented XSS vulnerabilities in VulnShop for educational and demonstration purposes.
1. Stored XSS - Product Reviews
Description

Stored XSS occurs when malicious JavaScript code is submitted by an attacker, stored in the database, and later rendered in the browser of other users.

Affected Endpoint
POST /reviews/product/<product_id>/add
Affected Template
templates/shop/product.html
Affected Model / Service

src/models/review.py
src/services/review_service.py
Vulnerable Code

<p class="review-text">
    {{ "{{ review.comment | safe }}" }}
</p>

The |safe filter disables Jinja2 automatic HTML escaping. Therefore, HTML and JavaScript submitted inside a review comment are rendered directly by the browser.

Example Payload
<script>alert('Stored XSS in VulnShop')</script>

Alternative payload:

<img src=x onerror="alert('Stored XSS')">
Automated Exploit Demonstration

Instead of manually submitting the payload through the review form, the vulnerability can also be exploited automatically using a Python script.

Exploit File

            exploits/Stored_XSS_exploit.py
Execution Command

            cd exploits
            python Stored_XSS_exploit.py
What the Script Does
  • Sends a POST request to the vulnerable review endpoint.
  • Injects a malicious JavaScript payload into a product review.
  • Stores the payload in the database.
  • The payload executes when a user visits the affected product page.
Target Endpoint

            POST /reviews/product/<product_id>/add
Example Request Generated by the Script

            reviewerName=XSS_Attacker
            rating=5
            comment=<img src=x onerror="alert('Stored XSS')">
Expected Result

After running the exploit script, opening the affected product page (for example /products/23) will trigger the injected JavaScript code because the application renders the review using the vulnerable |safe filter.

Impact
  • Execution of attacker-controlled JavaScript
  • Session cookie theft
  • Phishing attacks inside the application
  • Manipulation of page content
  • Attack affects every user opening the infected product page
Mitigation

Do not render user-generated content with |safe. Let Jinja2 escape the content automatically. 


<p class="review-text">
    {{ "{{ review.comment }}" }}
</p>

Additionally, user input should be validated and sanitized before storing it in the database.

Files Involved
File Purpose
src/models/review.py Stores malicious review payloads in the database.
src/services/review_service.py Business logic for creating and retrieving reviews.
src/routes/review_routes.py Receives the malicious POST request.
templates/shop/product.html Renders the malicious payload using the |safe filter.
exploits/Stored_XSS_exploit.py Automated exploit script used during the demonstration.
2. Reflected XSS - Product Search
Description

Reflected XSS occurs when user input is immediately included in the HTTP response without proper escaping. Unlike Stored XSS, the payload is not saved in the database. It is reflected directly from the request back into the page.

Affected Endpoint
GET /products/search?query=<payload>
Affected Template
templates/home.html
Affected Route
src/routes/product_routes.py
Vulnerable Code

{% raw %}
<strong>{{ search_query | safe }}</strong>
{% endraw %}

The value of search_query comes directly from the URL parameter query. Because it is rendered with the Jinja2 |safe filter, malicious HTML or JavaScript can be executed in the victim's browser.

Example Payload
<img src=x onerror="alert('Reflected XSS in VulnShop')">
Example Malicious URL

http://127.0.0.1:5000/products/search?query=%3Cimg%20src%3Dx%20onerror%3D%22alert%28%27Reflected%20XSS%20in%20VulnShop%27%29%22%3E
Manual Demonstration

The attacker creates a malicious URL containing the payload and sends it to a victim. When the victim opens the link, the payload is reflected into the search result page and executed by the browser.

Automated Exploit Demonstration

The exploit script generates a malicious search URL containing the encoded XSS payload. The generated URL can then be opened in the browser to trigger the vulnerability.

Exploit File
exploits/reflected_xss_exploit.py
Execution Command

cd exploits
python reflected_xss_exploit.py
What the Script Does
  • Creates a malicious XSS payload.
  • URL-encodes the payload.
  • Generates a vulnerable search URL.
  • The generated URL triggers the payload when opened in a browser.
Impact
  • The victim must open a malicious link.
  • JavaScript executes in the victim's browser.
  • The attacker can manipulate the page content.
  • The attacker can perform phishing inside the application.
  • The attacker may steal sensitive data if cookies are not protected.
Mitigation

Do not render reflected user input with |safe. Let Jinja2 escape the value automatically. 


{% raw %}
<strong>{{ search_query }}</strong>
{% endraw %}

Additionally, validate and encode user input before reflecting it in HTML responses.

Files Involved
File Purpose
src/routes/product_routes.py Receives the query URL parameter and passes it to the template.
templates/home.html Reflects the search query into the HTML response using |safe.
templates/base.html Contains the search form that sends the user input to the vulnerable endpoint.
exploits/reflected_xss_exploit.py Automated script used to generate the malicious reflected XSS URL.
