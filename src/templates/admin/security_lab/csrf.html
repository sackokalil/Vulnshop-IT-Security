Cross-Site Request Forgery (CSRF)

Security Lab: This page documents an intentionally implemented CSRF vulnerability in VulnShop for educational and demonstration purposes.
1. CSRF - Unauthorized Order Status Change
Description

Cross-Site Request Forgery occurs when an attacker tricks an authenticated user into sending an unwanted request to a web application.

In this vulnerability, an administrator can be tricked into opening a malicious page that silently sends a request to change the status of an order.

Affected Endpoint
POST /admin/orders/<order_id>/status
Affected Route
src/routes/order_routes.py
Affected Template
templates/admin/orders/order_detail.html
Affected Model / Service

src/models/order.py
src/services/order_service.py
Vulnerable Code

@admin_order_bp.route("/<int:order_id>/status", methods=["POST"])
def update_status(order_id):

    if "user_id" not in session:
        flash("Please login first.", "warning")
        return redirect(url_for("login.login_page"))

    if session.get("role") != "Admin":
        flash("Access denied.", "danger")
        return redirect(url_for("home.home_page"))

    status = request.form.get("status")

    # VULNERABILITY:
    # The application updates the order status without checking
    # whether the POST request contains a valid CSRF token.
    change_order_status(order_id, status)

    flash("Order status updated successfully.", "success")

    return redirect(url_for("admin_order.admin_order_detail", order_id=order_id))

The application verifies authentication and authorization, but it does not verify whether the request was intentionally submitted from the legitimate admin form.

Cross-Site Attack Architecture

Legitimate VulnShop application:
http://127.0.0.1:5000

Malicious attacker page:
http://127.0.0.1:8000/csrf_exploit.html

Flow:
Admin Browser
    |
    | opens malicious cute cat page
    v
http://127.0.0.1:8000/csrf_exploit.html
    |
    | hidden POST request with Admin session cookie
    v
http://127.0.0.1:5000/admin/orders/18/status
Manual Exploitation Scenario

1. The administrator logs into VulnShop:

   http://127.0.0.1:5000/admin/dashboard

2. The attacker prepares two files:

   cute_cat.bat
   csrf_exploit.html

3. The attacker convinces the administrator to open cute_cat.bat.

4. The batch file starts a local attacker server on:

   http://127.0.0.1:8000

5. The browser opens:

   http://127.0.0.1:8000/csrf_exploit.html

6. The administrator only sees a harmless cat picture.

7. In the background, the malicious HTML page submits:

   POST /admin/orders/18/status
   status=Paid

8. The browser automatically includes the active Admin session cookie.

9. VulnShop accepts the request and changes the order status.
Expected Vulnerable Result

If Order #18 changes its status without the administrator clicking the real "Update Status" button inside VulnShop, the CSRF vulnerability is successfully exploited.

Batch File Used for the Demo

@echo off
cd /d "%~dp0"

start "" http://127.0.0.1:8000/csrf_exploit.html

python -m http.server 8000

pause

The batch file changes into the exploit folder, opens the malicious HTML page in the browser, and starts a local HTTP server on port 8000. This makes the attacker page behave like a separate website.

Impact
  • Unauthorized order status modification
  • Abuse of an authenticated administrator session
  • Execution of sensitive actions without user consent
  • Loss of integrity in order management
  • Possible business logic manipulation
Mitigation

Sensitive state-changing requests must include a CSRF token that is generated by the server and verified on every POST request. 


import secrets
from flask import abort

@admin_order_bp.route("/<int:order_id>")
def admin_order_detail(order_id):

    session["csrf_token"] = secrets.token_hex(32)

    order, items = get_order_details(order_id)

    return render_template(
        "admin/orders/order_detail.html",
        order=order,
        items=items
    )


@admin_order_bp.route("/<int:order_id>/status", methods=["POST"])
def update_status(order_id):

    if "user_id" not in session:
        flash("Please login first.", "warning")
        return redirect(url_for("login.login_page"))

    if session.get("role") != "Admin":
        flash("Access denied.", "danger")
        return redirect(url_for("home.home_page"))

    submitted_token = request.form.get("csrf_token")
    session_token = session.get("csrf_token")

    if not submitted_token or submitted_token != session_token:
        abort(403)

    status = request.form.get("status")

    change_order_status(order_id, status)

    flash("Order status updated successfully.", "success")

    return redirect(url_for("admin_order.admin_order_detail", order_id=order_id))

Files Involved
File Purpose

templates/admin/orders/order_detail.html Contains the legitimate Admin form used to update an order status.
src/services/order_service.py Calls the model function that updates the order status.

templates/admin/security_lab/csrf.html This file. It documents the CSRF vulnerability and its mitigation.
Automated Exploit Demonstration

The exploit demonstration uses a harmless-looking cat page to hide the malicious request from the administrator. The administrator sees only the fake content, while the order status is changed in the background.

Exploit Files

exploits/cute_cat.bat
exploits/csrf_exploit.html
Execution Command

cd exploits
.\cute_cat.bat
What the Exploit Does
  • Starts a local attacker-controlled server on port 8000.
  • Opens the malicious page csrf_exploit.html.
  • Displays a harmless cat picture to the administrator.
  • Submits a hidden POST request to VulnShop on port 5000.
  • Uses the administrator's active session cookie automatically.
  • Changes Order #18 status to Paid.
Target Endpoint

POST http://127.0.0.1:5000/admin/orders/18/status
Expected Result

After the administrator opens the malicious file, the browser displays only a cat picture. In the background, the following request is sent: 


POST /admin/orders/18/status
status=Paid

If the order status changes to Paid without the administrator manually using the VulnShop status form, the CSRF attack is confirmed.
