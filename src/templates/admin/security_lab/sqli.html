SQL Injection

Security Lab: This page documents an intentionally implemented SQL Injection vulnerability in VulnShop for educational and demonstration purposes.
1. SQL Injection - Login Authentication Bypass
Description

SQL Injection occurs when user input is inserted directly into a SQL query without proper parameterization. An attacker can inject SQL code and change the meaning of the original query.

In this vulnerability, the login form is intentionally vulnerable. By injecting SQL into the email field, an attacker can bypass authentication and log in as an existing user, for example an administrator.

Affected Endpoint
POST /login
Affected Route
src/routes/auth_routes.py
Affected Service
src/services/auth_service.py
Affected Template
templates/auth/login.html
Vulnerable Code

def authenticate_user(email, password):

    conn = get_db_connection()

    query = (
        "SELECT id, first_name, last_name, username, email, password, role, status, created_at "
        "FROM users "
        f"WHERE email = '{email}' AND password = '{password}'"
    )

    user = conn.execute(query).fetchone()

    conn.close()

    return user

The vulnerability is caused by the direct insertion of email and password into the SQL query using an f-string. The database receives the injected SQL as part of the query.

Example Payload
' OR role='Admin' AND status='Active' --

The password can contain any value because the SQL comment operator -- comments out the remaining password check.

Generated SQL Query

SELECT id, first_name, last_name, username, email, password, role, status, created_at
FROM users
WHERE email = '' OR role='Admin' AND status='Active' -- ' AND password = 'anything'

Because the injected condition searches for an active administrator, the database returns an admin user. Flask then creates a real session for that user.

Manual Exploitation Scenario

1. Open the login page:

   /login

2. Enter the following payload in the email field:

   ' OR role='Admin' AND status='Active' -- 

3. Enter any password, for example:

   anything

4. Submit the login form.
Expected Vulnerable Result

If the application is vulnerable, the attacker is redirected to the admin dashboard because the SQL query returns an active administrator. 


/admin/dashboard

This confirms that the attacker bypassed authentication and obtained a valid Flask session as an admin user.

Impact
  • Authentication bypass
  • Unauthorized login as another user
  • Privilege escalation if the returned user is an administrator
  • Creation of a valid server-side session for the attacker
  • Unauthorized access to the admin dashboard
  • Possible exposure or manipulation of sensitive data
Mitigation

User input must never be inserted directly into SQL queries. The secure solution is to use parameterized queries. 


def authenticate_user(email, password):

    conn = get_db_connection()

    query = """
        SELECT id, first_name, last_name, username, email, password, role, status, created_at
        FROM users
        WHERE email = ?
        AND password = ?
    """

    user = conn.execute(query, (email, password)).fetchone()

    conn.close()

    return user

With parameterized queries, the database treats user input as data, not as executable SQL code.

In a real production system, passwords should also be hashed and verified using a secure password hashing function instead of being stored in plain text.

2. Union-Based SQL Injection - Product Search

Union-Based SQL Injection occurs when user input is inserted directly into an SQL query and allows an attacker to combine the original query with another query using the SQL UNION operator.

In VulnShop, the product search feature intentionally contains a vulnerable query that can be exploited to retrieve data from the users table.

Affected Endpoint

            GET /products/search?query=
Affected Files

            src/routes/product_routes.py
            src/models/product.py
            src/services/product_service.py
            templates/home.html
Vulnerable Query

            query = (
            "SELECT id, name, description, price, category, image_url "
            "FROM products "
            f"WHERE name LIKE '%{keyword}%' "
            f"OR description LIKE '%{keyword}%' "
            f"OR category LIKE '%{keyword}%' "
            "ORDER BY id DESC"
            )

The value of keyword is inserted directly into the SQL query. An attacker can therefore inject additional SQL statements.

Attack Payload

            ' UNION SELECT
            id,
            email,
            password,
            0,
            role,
            'uploads/products/default.jpg'
            FROM users --
Column Mapping
Injected Value Displayed As
email Product Name
password Product Description
role Product Category
0 Product Price
Expected Vulnerable Result

Instead of returning only products, the application displays records from the users table as fake products.

Sensitive information such as email addresses, passwords and user roles becomes visible inside the product list.

Automated Exploit Demonstration
Exploit File

            exploits/sqli_union_exploit.py
Execution Command

            cd exploits
            python sqli_union_exploit.py
What the Script Does
  • Opens the VulnShop website in a browser.
  • Automatically inserts a UNION SELECT payload into the search field.
  • Submits the search form.
  • Triggers the vulnerable SQL query.
  • Displays user records as fake products.
Requirements
  • Google Chrome must be installed.
  • Selenium must be installed.
  • ChromeDriver must match the installed Chrome version.
Impact
  • Disclosure of user accounts.
  • Disclosure of passwords stored in the database.
  • Disclosure of user roles.
  • Unauthorized access to sensitive information.
  • Violation of confidentiality.
Mitigation

All database queries should use parameterized statements instead of concatenating user input directly into SQL strings. 


            products = conn.execute("""
            SELECT id, name, description, price, category, image_url
            FROM products
            WHERE name LIKE ?
            OR description LIKE ?
            OR category LIKE ?
            """, (
            f"%{keyword}%",
            f"%{keyword}%",
            f"%{keyword}%"
            ))
