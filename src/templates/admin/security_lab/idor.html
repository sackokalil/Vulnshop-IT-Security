Insecure Direct Object Reference (IDOR)

Security Lab: This page documents an intentionally implemented IDOR vulnerability in VulnShop for educational and demonstration purposes.
1. IDOR - Unauthorized Order Access
Description

IDOR occurs when an application exposes a direct object identifier, such as an order ID, user ID, or file ID, and does not verify whether the authenticated user is allowed to access that object.

In this vulnerability, a logged-in user can manually modify the order_id in the URL and access another user's order details.

Affected Endpoint
GET /orders/<order_id>
Affected Route
src/routes/order_routes.py
Affected Template
templates/shop/order_detail.html
Affected Model / Service

src/models/order.py
src/services/order_service.py
Vulnerable Code

@order_bp.route("/<int:order_id>")
def order_detail(order_id):

    # The application checks if the user is logged in.
    if "user_id" not in session:
        flash("Please login first.", "warning")
        return redirect(url_for("login.login_page"))

    # The order_id comes directly from the URL.
    # Example:
    # /orders/1
    # /orders/2
    order, items = get_order_details(order_id)

    # VULNERABILITY:
    # The application does not check if this order belongs
    # to the currently logged-in user.
    #
    # Missing check:
    # if order["user_id"] != session["user_id"]:
    #     abort(403)

    return render_template(
        "shop/order_detail.html",
        order=order,
        items=items
    )

The application verifies authentication, but not authorization. This means the user must be logged in, but the application does not verify whether the requested order belongs to that user.

Manual Exploitation Scenario

1. Login as User A
2. Create an order
3. Visit: /orders  to see user A's order_id

4. Logout
5. Login as User B

6. While logged in as User B, manually change the URL to:

   /orders/user_A_order_id
Expected Vulnerable Result

If User B can see Order #user_A_order_id, which belongs to User A, then the IDOR vulnerability is successfully exploited.

Impact
  • Unauthorized access to another user's order details
  • Exposure of sensitive customer information
  • Violation of confidentiality
  • Possible data leakage through predictable object IDs
  • Broken authorization logic
Mitigation

The application must verify that the requested object belongs to the currently authenticated user before displaying it. 


from flask import abort

@order_bp.route("/<int:order_id>")
def order_detail(order_id):

    # Check authentication.
    if "user_id" not in session:
        flash("Please login first.", "warning")
        return redirect(url_for("login.login_page"))

    # Get the requested order.
    order, items = get_order_details(order_id)

    # Return 404 if the order does not exist.
    if not order:
        abort(404)

    # SECURITY FIX:
    # Check authorization.
    # The logged-in user must be the owner of the requested order.
    if order["user_id"] != session["user_id"]:

        create_security_event(
            event_type="IDOR_BLOCKED",
            severity="High",
            description=f"Blocked IDOR attempt: User {session['user_id']} tried to access order {order_id} owned by user {order['user_id']}.",
            user_id=session["user_id"],
            endpoint=request.path,
            ip_address=request.remote_addr,
            user_agent=request.headers.get("User-Agent")
        )

        abort(403)

    return render_template(
        "shop/order_detail.html",
        order=order,
        items=items
    )

Automated Exploit Demonstration

Instead of manually modifying the URL in the browser, the vulnerability can also be exploited automatically using a Python script.

Exploit File

            exploits/idor_exploit.py
Execution Command

            cd exploits
            python idor_exploit.py
What the Script Does
  • Logs in as a legitimate user.
  • Maintains the authenticated session.
  • Requests another user's order by manipulating the order ID.
  • Checks whether the vulnerable application returns the order details.
  • Confirms the IDOR vulnerability if unauthorized data is exposed.
Target Endpoint

            GET /orders/<order_id>
Example Configuration

            ATTACKER_EMAIL = "user@test.com"
            ATTACKER_PASSWORD = "123456"

            VICTIM_ORDER_ID = 12
Expected Result

After running the exploit script, the attacker is authenticated as user@test.com and attempts to access /orders/12.

If the application returns Order #12 even though it belongs to another user, the script prints: 


            [+] Victim order page is accessible.
            [+] IDOR vulnerability confirmed.
            [+] The attacker accessed an order that does not belong to them.

Additionally, the attack can be recorded in the security_events table and displayed in the Recent Security Events dashboard section.
